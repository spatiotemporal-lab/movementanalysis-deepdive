Movement Analysis I

Per-Point Transportation Mode Prediction with GPS Trajectories

Author

Chantal Fuchs

Published

April 29, 2026

Introduction

Understanding how people move through cities is a core problem in urban planning and transportation research. GPS trajectories from smartphones capture rich movement data, but they don’t directly tell us how someone is moving — whether they’re walking, cycling, or sitting on a bus. This label has to be inferred from the movement itself.

In this assignment I work with the GeoLife dataset (Zheng et al., 2011), a collection of GPS trajectories from 182 users recorded in Beijing between 2007 and 2012. A labelled subset of 73 users includes transportation mode annotations — walking, cycling, bus, car, subway, and train. The task is to build a model that can predict the transportation mode for each individual GPS point in the unlabelled validation dataset.

This is a harder version of what we did in Advanced Environmental Statistics: instead of summarising features per segment and predicting one label per track, here every point needs its own prediction. That means the features have to capture local movement behaviour at the point level — which is actually much more interesting.

Step 2: Feature Engineering

Feature engineering is where most of the work happens. Raw GPS points just have coordinates and a timestamp — everything else has to be computed. I derive a set of movement parameters that capture different aspects of how a person is moving at each point.

Base parameters: speed and time lag

Step length (distance between consecutive points) and time lag (time between consecutive timestamps) are the fundamental building blocks. Speed follows directly.

Code 
full_dataset <- full_dataset |>
  mutate(
    steplength = as.numeric(st_distance(lead(geom), geom, by_element = TRUE)),
    timelag    = as.numeric(difftime(lead(datetime), datetime, units = "secs")),
    speed      = steplength / timelag,
    .by = track_id
  )

Acceleration

Acceleration captures whether a person is speeding up or slowing down. Walking and cycling have much more variable acceleration than, say, taking a train.

Code 
full_dataset <- full_dataset |>
  mutate(
    acceleration = (lead(speed) - speed) / timelag,
    .by = track_id
  )

Turning angle

The turning angle between consecutive movement steps is another useful signal. Pedestrians and cyclists tend to make more frequent direction changes than vehicles following roads or rail lines. I compute the bearing of each step and then the absolute angular difference between consecutive steps.

Code 
# Extract coordinates for bearing calculation
coords <- st_coordinates(full_dataset)

full_dataset <- full_dataset |>
  mutate(
    x = coords[, 1],
    y = coords[, 2]
  )

full_dataset <- full_dataset |>
  mutate(
    dx        = lead(x) - x,
    dy        = lead(y) - y,
    bearing   = atan2(dx, dy),          # radians
    turn_angle = abs(bearing - lag(bearing)),
    # Wrap to [0, pi]
    turn_angle = ifelse(turn_angle > pi, 2 * pi - turn_angle, turn_angle),
    .by = track_id
  )

Sinuosity (two window sizes)

Sinuosity — the ratio of actual path length to straight-line distance — measures path curvature. I compute it over two different window sizes (5 and 10 points) to capture both local and slightly broader movement patterns.

Code 
full_dataset <- full_dataset |>
  mutate(
    # 5-point window
    straight_dist5 = as.numeric(st_distance(lead(geom, 5), geom, by_element = TRUE)),
    full_dist5     = rollsum(steplength, 5, fill = NA, align = "left"),
    sinuosity5     = full_dist5 / straight_dist5,

    # 10-point window
    straight_dist10 = as.numeric(st_distance(lead(geom, 10), geom, by_element = TRUE)),
    full_dist10     = rollsum(steplength, 10, fill = NA, align = "left"),
    sinuosity10     = full_dist10 / straight_dist10,

    .by = track_id
  )

Speed distribution check

Before training, it’s worth looking at how speed distributes across modes. This confirms that speed alone already carries a lot of discriminative power.

Code 
full_dataset |>

  st_drop_geometry() |>
  filter(speed < 30, speed > 0) |>
  mutate(mode = fct_reorder(mode, speed, .fun = median, na.rm = TRUE)) |>
  ggplot(aes(speed, mode, fill = mode)) +
  geom_boxplot(outlier.size = 0.3, alpha = 0.8) +
  scale_fill_brewer(palette = "Set2") +
  labs(
    x = "Speed (m/s)",
    y = "Transportation mode",
    title = "Per-point speed by transportation mode"
  ) +
  theme_minimal() +
  theme(legend.position = "none")

Walking and cycling cluster at low speeds, while car, bus, and train spread across higher values. Subway is interestingly variable — underground sections have consistent high speed, but stops bring it to zero. This overlap between modes is exactly why we need multiple features, not just speed alone.

Step 3: Prepare Data for Modelling

For per-point prediction I work directly with the point-level features — no aggregation. I select the relevant columns, remove rows with NA values in the features (these arise at segment boundaries due to lead()/lag() and rolling windows), and split back into training, testing, and validation sets.

Code 
feature_cols <- c("speed", "acceleration", "turn_angle",
                  "sinuosity5", "sinuosity10", "steplength", "timelag")

# Point-level dataset (drop geometry, keep only needed columns)
points_df <- full_dataset |>
  st_drop_geometry() |>
  select(data, track_id, mode, all_of(feature_cols))

# Remove rows where any feature is NA or infinite
points_df <- points_df |>
  filter(if_all(all_of(feature_cols), ~ !is.na(.) & is.finite(.)))

# Split (NA-removed versions for training/testing)
points_training   <- points_df |> filter(data == "training")   |> select(-data, -track_id) |> mutate(mode = factor(mode))
points_testing    <- points_df |> filter(data == "testing")     |> select(-data, -track_id) |> mutate(mode = factor(mode))
points_validation <- points_df |> filter(data == "validation")  |> select(-data, -track_id)

  }))

Step 4: Train a Random Forest

I use a Random Forest via the ranger package. Random Forests handle the class imbalance and feature interactions in this dataset much better than a single decision tree — which is important here because modes like train and subway are quite rare compared to walk.

  formula        = mode ~ .,
  data           = points_training,
  num.trees      = 300,
  importance     = "impurity",
  num.threads    = 2,

Discussion

The Random Forest model reaches an overall accuracy of 61.7 % on the per-point testing data, compared to 54.2 % for the single CART on the same task (see comparison table). Both are below the ~69 % the CART baseline achieved in the Advanced Environmental Statistics assignment — but that comparison isn’t fair. The AES CART predicted one label per segment using aggregated features (mean, max, min), which is a much easier task. Here every single GPS point gets its own prediction from raw local features, with no smoothing over a whole trajectory. The Random Forest’s advantage over the single tree is most visible for rare classes like train and subway, where the ensemble averaging stabilises predictions that a single tree gets wrong.

The model struggles most with bus vs. car and subway vs. train, which is visible in the confusion matrix. That’s not particularly surprising: from a kinematics perspective, sitting in a bus and sitting in a car feel almost identical. Both move at similar speeds along roads, neither produces a distinctive turning-angle signature, and the stops of a bus are too brief to register clearly in a 5–10 point sinuosity window. Distinguishing them properly would require context — road network type, stop frequency, distance to rail — which is exactly what the upcoming CMA2 lecture on context-aware movement analysis addresses.

Walking is the best-predicted class with a sensitivity of ~81 %, while cycling reaches ~63 %. The gap between them reflects the value of adding turning angle and dual-window sinuosity on top of speed — pedestrians genuinely do take more winding paths and change direction more often than cyclists, and those features capture that difference.

The OOB training error of ~34.5 % is a few points better than the ~38.3 % test error, which means the model generalises reasonably — it’s not dramatically overfitting. The main weak point is the rare classes: train has a sensitivity of only 31 % because it’s so underrepresented in training. Giving those classes higher weight or using oversampling (SMOTE) would be a logical next step.

For the final validation predictions, boundary points where rolling-window features (sinuosity, acceleration) were undefined due to segment edges are imputed with training medians. This ensures all 240’449 validation points receive a prediction rather than being silently dropped.
